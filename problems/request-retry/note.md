# 手写请求重试机制

## 场景

请求失败后自动重试 N 次，并使用指数退避控制每次重试前的等待时间。

常见使用场景：

- 网络抖动导致请求偶发失败
- 后端服务短暂不可用
- 某些接口需要容忍临时错误

## 题目目标

实现一个 `requestRetry` 方法：

- 接收一个请求函数 `requestFn`
- 请求成功时，直接返回成功结果
- 请求失败时，最多自动重试 `retries` 次
- 每次重试前等待一段时间，等待时间按指数递增
- 如果全部失败，返回最后一次失败的错误

## 函数签名

```js
function requestRetry(requestFn, retries = 3, delay = 0) {}
```

## 核心思路

把“执行请求”包装成一个可重复调用的流程：

1. 执行 `requestFn`
2. 成功则 `resolve`
3. 失败则判断还有没有剩余重试次数
4. 有剩余次数：按指数退避计算等待时间
5. 没有剩余次数：抛出最后一次错误

当前实现中的退避公式：

```js
const wait = delay * 2 ** count;
```

如果 `delay = 500`，每次失败后的等待时间依次是：

- 第 1 次重试前等待 `500ms`
- 第 2 次重试前等待 `1000ms`
- 第 3 次重试前等待 `2000ms`

## 关键点

- `requestFn` 必须是函数，不能直接传一个 Promise，因为 Promise 一旦创建就已经开始执行，无法重新发起。
- 重试次数通常表示“失败后额外再试几次”，所以总请求次数是 `1 + retries`。
- 每一次重试都需要重新调用 `requestFn()`。
- 指数退避可以避免请求失败后立刻高频重试，减轻服务端压力。
- `count` 既可以用来判断是否超过最大重试次数，也可以用来计算当前等待时间。
- 如果 `delay` 为 `0`，指数退避结果也会一直是 `0`，相当于不等待直接重试。

## 执行流程

1. 初始化失败次数 `count = 0`。
2. 调用 `attempt()` 发起第一次请求。
3. 请求成功：直接 `resolve`。
4. 请求失败：判断 `count >= retries`。
5. 如果已经达到最大重试次数，直接 `reject(err)`。
6. 如果还能继续重试，计算 `wait = delay * 2 ** count`。
7. `count++`，然后通过 `setTimeout(attempt, wait)` 延迟发起下一次请求。

## 可选扩展

- 支持只对特定错误重试
- 支持 `onRetry` 回调，记录当前第几次重试
- 支持 `AbortController` 取消重试流程
- 支持最大等待时间上限，避免退避时间过长
